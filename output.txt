1. Project Overview & Problem Statement
Cascade predicts whether a user will click on an ad — in the few milliseconds before the ad is shown. This is one of the highest-value problems in all of tech: it's how Google, Meta, and every ad platform decide which ad to show you, and it directly drives billions in revenue. When you load a webpage, an auction happens in real time to decide which ad appears. To bid intelligently, the platform must instantly estimate: ‘how likely is THIS user to click THIS ad?’ That estimate is the Click-Through Rate (CTR), and predicting it well is the difference between profit and loss.
This is positioned as a serious, industrial machine-learning project — not a toy classifier. The depth comes from the three things that make CTR prediction genuinely hard in the real world: extreme class imbalance (almost nobody clicks — maybe 1 in 100), massive scale with the need for instant predictions, and the fact that good feature engineering matters far more than the model choice. Each is a dedicated section in this document.
In plain words
Cascade is the brain behind ‘which ad do I show this person?’ It has to be smart (predict clicks accurately), fast (answer in a millisecond), and honest about a hard reality — clicks are incredibly rare, so the usual ML tricks fail and you need the right techniques. That combination is what makes it an industrial-grade project, not a tutorial.

2. The Core Idea in Plain English
At its heart, Cascade answers one yes/no question with a probability: ‘will this user click this ad?’ The output isn't just ‘yes’ or ‘no’ — it's a probability like 0.03 (a 3% chance of clicking). That number is what the ad system uses to decide how much to bid.
The challenge that shapes everything is that clicks are extremely rare. Out of 100 ads shown, maybe 1 gets clicked. This sounds like a small detail but it breaks naive machine learning completely — because a lazy model can be 99% ‘accurate’ by simply predicting ‘no click’ every single time, while being totally useless. Handling this rarity correctly is the soul of the project.
Plain analogy
Imagine trying to predict which lottery tickets will win when almost none do. If you just say ‘this ticket will lose’ every time, you're right 99.9% of the time — and completely worthless, because you never find the winner. CTR prediction is the same: being ‘accurate’ by always saying ‘no click’ is a trap. The whole skill is finding the rare clicks.
3. System Architecture — The Pipeline
Cascade is a pipeline with two sides: an offline TRAINING side that builds the model from historical data, and an online SERVING side that uses the trained model to make instant predictions. This split is fundamental — the heavy learning happens once, offline; the fast predicting happens millions of times, online.

5. The Data & The Imbalance Problem
Cascade learns from historical ad logs — records of ads shown and whether each was clicked. Each row has information about the user, the ad, and the context (time, device, page), plus the answer: clicked (1) or not (0).
Dataset: Criteo or Avazu CTR data — famous, free, real-world ad datasets with millions of rows. Industrial-scale data, publicly available.
The defining feature: imbalance — only around 1–3% of rows are clicks. The other 97–99% are non-clicks.
5.1 Why Imbalance Breaks Naive Models
If you train a normal model on this data and measure accuracy, it learns the laziest possible trick: always predict ‘no click.’ That's 98% accurate and completely useless. The model never learns what a click looks like, because ignoring the rare clicks barely hurts its accuracy score.
5.2 How We Fix It
Class weights — we tell the model that getting a rare click wrong is much more costly than getting a common non-click wrong, forcing it to pay attention to clicks.
Smart sampling — optionally balance the training data so clicks aren't drowned out.
The right metric — most importantly, we STOP using accuracy and use metrics built for imbalance (Section 10).
The interview gold here
‘My data was 99% non-clicks, so accuracy was meaningless — a model predicting ‘no’ always scored 99%. I used class weights and evaluated with AUC and PR-AUC instead.’ That single explanation instantly marks you as someone who has handled real-world data, not just clean tutorials.
6. Feature Engineering — The Real Battleground
This is where CTR prediction is actually won. The raw data (user ID, ad ID, timestamp) is weak on its own. The magic is engineering features — creating powerful new signals from the raw data. This matters far more than which model you pick.
Historical click rates — the single most powerful feature type: ‘how often has this user clicked before?’, ‘how often does this ad get clicked?’, ‘how often do users click this ad CATEGORY?’ Past behaviour strongly predicts future clicks.
Cross features — combinations like ‘this user type + this ad type’ — capturing that sports fans click sports ads. The interaction matters more than either alone.
Time features — hour of day, weekday vs weekend; click behaviour changes with time.
Context features — device, page type, ad position on screen (top ads get more clicks).
Handling high-cardinality IDs — there are millions of unique user and ad IDs. We encode them cleverly (e.g. target encoding) instead of naively, which would explode into millions of useless columns.
Why this section wins interviews
When asked ‘how did you improve your model?’, the weak answer is ‘I tried a bigger model.’ The strong answer is ‘I engineered historical click-rate features for the user and ad, which gave the biggest lift — far more than any model change.’ Knowing that features beat models is a core sign of real ML experience.
7. The Modeling Ladder — Baseline to Gradient Boosting
As always in serious ML, we climb a ladder — start simple, prove each step earns its place.
Baseline — Logistic Regression. A simple, fast, interpretable model. It sets the score to beat and often does surprisingly well on CTR.
Gradient Boosting — XGBoost / LightGBM. The workhorse champions for tabular data. They build many small decision trees that each fix the previous ones' mistakes, capturing complex patterns the baseline misses. This is usually the winner.
Compare honestly. Score every model the same way on the same unseen data, and report a clear table proving the gradient-boosting model beat the baseline by a meaningful margin.
What gradient boosting is (plain analogy)
Gradient boosting is like a team of students taking turns on a hard problem: the first gives a rough answer, the second focuses only on fixing the first's mistakes, the third fixes what's still wrong, and so on. Each one specialises in the leftover errors. Together, this ‘team of correctors’ becomes extremely accurate — which is why XGBoost and LightGBM dominate tabular ML.
8. The LSTM Module — Learning from User History
Gradient boosting is brilliant but treats each ad request fairly independently. Yet a user's recent behaviour is a SEQUENCE that tells a story: someone who just browsed three laptops in a row is primed to click a laptop ad. To capture this ordered history, we add an LSTM module.
8.1 What an LSTM Is, Simply
An LSTM (Long Short-Term Memory) is a type of neural network built to learn from sequences — data where ORDER matters. It reads a sequence step by step while keeping a ‘memory’ of what it has seen, so earlier events can influence its understanding of later ones.
Plain analogy
An LSTM is like reading a story and remembering the plot as you go. By the last page, you understand a sentence based on everything that happened before. For Cascade, the ‘story’ is a user's recent activity — browsed phones, then cases, then chargers — and the LSTM ‘reads’ this to sense they're in shopping mode and likely to click a related ad. A flat model, reading each event in isolation, misses this momentum.
What it adds — a signal capturing the user's recent behavioural pattern, which we feed alongside the other features.
Why include it — it demonstrates both classical ML (gradient boosting) AND deep learning (LSTM/sequences) in one project, and captures real predictive signal the boosting model alone would miss.
9. Hyperparameter Tuning & Cross Validation
Two essential discipline steps that turn a rough model into a reliable one.
9.1 Cross Validation — Trusting Your Score
Instead of testing the model on just one slice of data (which might be lucky or unlucky), cross-validation tests it on several different slices and averages the results — giving a far more trustworthy estimate of true performance. For CTR's time-based data, we use a time-aware split so we never train on the future to predict the past.
Plain analogy for cross-validation
Judging a student on a single exam is risky — they might have had a good or bad day. Cross-validation is like giving several exams and averaging the scores: a much fairer, more reliable measure of how good they really are.
9.2 Hyperparameter Tuning — Finding the Best Settings
Models like XGBoost have dials (hyperparameters) — how many trees, how deep, how fast to learn. The right settings make a big difference. Instead of guessing, we use Optuna to search intelligently for the best combination, guided by cross-validation scores. This is systematic, not trial-and-error.
10. Honest Evaluation — The Right Metrics for Clicks
Because of the imbalance, picking the right metric isn't a detail — it's the difference between a real evaluation and a fooled one.
Metric
Plain-English meaning
Why it fits CTR
AUC-ROC
How well the model ranks clickers above non-clickers.
Doesn't care about the imbalance — measures ranking ability, which is what bidding needs.
PR-AUC
How well it finds the rare clicks specifically.
Designed for rare-event problems — the honest score for imbalanced data.
Log Loss
How good the predicted PROBABILITIES are, not just yes/no.
Bidding uses the probability itself, so its quality matters.
Calibration
When it says ‘10% chance,’ do ~10% actually click?
Critical — the platform bids real money based on the probability being honest.
Calibration deserves special mention: in ad bidding, the platform literally bets money proportional to the predicted probability. If the model says ‘10% chance’ but really it's 2%, the platform overbids and loses money. So we don't just need the ranking to be right — we need the probabilities themselves to be honest. Caring about calibration is a senior-level signal.
11. The Serving Layer — Predicting in a Millisecond
A CTR model is useless if it's slow — the ad auction waits for no one. This layer makes Cascade real-time, covering the production/inference keywords.
FastAPI service — receives a user+ad request and returns a click probability almost instantly.
Speed first — LightGBM is chosen partly because it predicts extremely fast; features are pre-computed and cached so we're not calculating them from scratch on every request.
Docker — packages the model and code to run identically anywhere.
The latency point — in real ad systems, the whole prediction must happen in a few milliseconds. Being able to discuss this constraint — and why you chose a fast model and cached features — shows you understand production ML, not just training.

13. Scalability, Trade-offs & Decisions
13.1 Scalability Notes
Huge data — LightGBM is memory-efficient and handles millions of rows well; for truly massive data it can train on samples or in chunks.
Fast features at serving — the most predictive features (historical click rates) are pre-computed and stored, so serving just looks them up rather than recalculating.
Retraining — click patterns drift (new ads, trends), so the model is retrained regularly on fresh data, tracked with MLflow.

14. Challenges Faced & How They Were Solved
Challenge 1 — A 99%-accurate model that found zero clicks
The first model proudly hit 99% accuracy — then we realised it was simply predicting ‘no click’ for everything, because clicks were only 1% of the data. It was useless. Fix: switched the metric to AUC and PR-AUC, and added class weights so the model was forced to take rare clicks seriously. The right-metric-for-imbalanced-data lesson is one of the most valuable stories you can tell in an ML interview.
Challenge 2 — The model barely beat the simple baseline
Early on, XGBoost only marginally beat plain logistic regression, which raised the question of whether the complexity was worth it. Fix: this pushed us back into feature engineering (Section 6), where adding historical click-rate features gave a large jump. The lesson — when stuck, improve features before blaming the model — is pure ML maturity, and the single most useful thing to say about how you improved results.
Challenge 3 — Honest-looking probabilities that lost money
The model ranked clicks well (good AUC) but its probabilities were off — it would say ‘10%’ when the real rate was 3%. For ad bidding, that means overbidding and losing money. Fix: added calibration so the predicted probabilities matched reality. Understanding that ranking ISN'T enough — the probability itself must be trustworthy — is a senior-level insight most candidates miss.
Challenge 4 — Accurate but too slow to use
An early version computed features on the fly at prediction time, making each prediction far too slow for a real-time auction. Fix: pre-computed and cached the heavy features (like historical click rates) so serving just looks them up, and chose LightGBM for its fast predictions. The model went from accurate-but-unusable to accurate-and-instant — exactly the production thinking that defines an ML engineer.
